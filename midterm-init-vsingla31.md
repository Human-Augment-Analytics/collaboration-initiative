# Part I - Initiative Report

## HAAG Project Snapshot + Starter Path

### Initiative Overview

This initiative focuses on creating a lightweight onboarding and project-clarity process for HAAG. The idea is simple: each project should have one clear, current starting point for newcomers and potential contributors.

At the moment, project information is often scattered across Slack threads, GitHub repositories, shared documents, webpages, and older materials. Because of that, it can be difficult for someone new to quickly understand what a project is about, what it is currently working toward, who is responsible for what, and how they can begin contributing.

To address that, the initiative proposes two simple artifacts for each project:

- **Project Snapshot:** a short, one-page summary of the project
- **Demo Log:** a lightweight running record of meaningful project updates

The Project Snapshot is meant to quickly cover the basics:

- what the project is
- what problem it is trying to solve
- what the current goal is for the month or semester
- the project’s current status
- who the main contacts are
- which links matter most
- how a newcomer can get started

The Demo Log is intended to capture progress over time in a simple, usable format. Each entry records what changed, why it matters, a link to the related artifact, and the next step or open question.

The process is designed to be asynchronous and low-overhead. Instead of depending on a live walkthrough or a long onboarding conversation each time someone joins, the team can point people to one reliable place to begin. The broader goal is to reduce onboarding friction, improve visibility into project work, and make it easier for more people to contribute.

At this stage, I see the initiative as an organizational experiment in the discovery-to-pilot phase rather than a fully implemented system. My goal this semester is to define the procedure clearly, identify a realistic way to test it, and move it toward a small pilot.

### Current Hypotheses and KPIs

The main hypothesis behind this initiative is that if HAAG projects provide a simple, standardized Project Snapshot and Demo Log, newcomers and potential contributors will understand project context more quickly, ask fewer repeated onboarding questions, and feel more confident about how to get started.

A related hypothesis is that if project context is easier to find and stays more current, project teams will spend less time repeating introductory information and more time doing actual project work.

The KPIs identified so far are:

- **Clarity score:** a self-reported score, such as 1–5, on whether someone understands what the project is doing and how they could contribute
- **Time-to-context:** how long it takes a newcomer to find enough information to understand where to begin
- **Repeated onboarding questions:** how often project leads or managers receive the same introductory questions, such as where the repo is, who to contact, or how to get started
- **Starter-task readiness:** whether a newcomer can identify at least one reasonable next step after reviewing the snapshot

At this point, these KPIs are identified but not yet fully measured. Most of the work so far has focused on defining the problem and selecting indicators that are realistic and useful. I have not yet run a before-and-after pilot with enough data to make a claim about improvement.

The remaining work includes:

- collecting a small baseline from current onboarding experiences
- piloting the snapshot with one project or team
- gathering quick user feedback
- comparing the before-and-after experience using the KPIs above

What I especially want to validate is whether this approach actually reduces confusion and shortens the time it takes for someone new to get oriented.

### Pilot Testing Approach

The testing plan is centered on a small pilot rather than a broad rollout. The idea is to start with one willing project, create the Snapshot and Demo Log for that team, and then ask a small number of users to interact with them and share feedback.

The testing process would look like this:

- identify where the project’s current context is scattered
- draft a Project Snapshot and Demo Log template
- fill in the templates for one pilot project
- ask a small group of newcomers or potential contributors to review them
- ask a few quick questions about clarity, contacts, and how to begin
- compare that experience to the current status quo
- revise the procedure based on what was unclear or missing

In simple terms, the flow is:

**scattered onboarding experience → template creation → pilot with one project → small user review → feedback collection → revision → possible broader use**

### Stakeholder Engagement

The key stakeholders for this initiative include project leads, project managers or co-managers, newcomers, current researchers, and HAAG leadership. Each of these groups is connected to the problem in a different way.

Newcomers and potential contributors are the primary audience because they experience onboarding confusion most directly. Project leads and managers matter because they are often the ones answering the same introductory questions or redirecting people to resources. HAAG leadership may also have an interest in the initiative because it could create more visibility and consistency across projects.

So far, stakeholder engagement has been more indirect than formal. My understanding of the problem has come mainly from observing how project information is currently spread out and thinking through the friction points that come up when someone needs project context quickly. That means the initiative has not yet reached the stage of a formal pilot with committed participants.

This does not fully match the ideal scenario. In a stronger version of the project, I would already have one pilot team and some early stakeholder feedback. Instead, the initiative is still in the stage of refining the procedure and defining the smallest useful version of the experiment.

Given that, I think the right move is to keep the scope very small. Rather than trying to involve multiple projects at once, the next step should be to find one willing team and test the simplest workable version of the process. I also think it is important to frame this as a way to reduce overhead, not add bureaucracy.

### Sustainability and Documentation

The main sustainability goal is to make the process simple enough that teams can realistically keep it current over time. So far, I have started documenting:

- the problem statement
- the purpose of the Project Snapshot
- the purpose of the Demo Log
- the suggested content for each artifact
- the idea of including a “Last Updated” date
- the idea of assigning both a primary owner and a backup owner

My current plan is for the process to live in a GitHub repository or wiki-style location where teams can link to it easily and update it asynchronously. A likely structure would be:

```text
PROJECT_SNAPSHOT.md
DEMO_LOG.md or a demos/ folder
a simple index page linking to the latest project snapshot
```

The additional documentation I still plan to complete includes:

- a reusable Project Snapshot template
- a reusable Demo Log template
- a short note on ownership and update expectations
- a small pilot procedure explaining how to test the initiative with one project

### Current Progress Tracking

At this point, I am measuring progress more in terms of design progress and pilot readiness than final outcome data.

The main signs of progress so far are:

- I narrowed the initiative from a broader collaboration idea to a more realistic onboarding and project-clarity approach
- I clearly defined the problem the initiative is trying to solve
- I identified a small set of realistic KPIs
- I outlined the core artifacts the initiative would require
- I defined a small pilot strategy instead of an overly broad rollout

The main success indicators I am watching for are whether the initiative becomes simple enough to test, useful enough to reduce confusion, and lightweight enough that a real team would actually adopt it.

The main warning signs are the opposite. If the materials become too long, too formal, or too difficult to keep updated, the initiative will lose practical value.

### Obstacles and Bottlenecks

One challenge has been scope. I originally considered a broader collaboration-focused initiative, but that version had too many moving parts and would have been difficult to test within a limited timeline. Narrowing the focus to onboarding and project clarity made the work much more manageable.

Another challenge has been the gap between a good idea in theory and a manageable experiment in practice. It is easy to describe a system that sounds useful. It is harder to define the smallest version that a real team would actually be willing to try.

A few likely challenges have already become clear:

- information staleness is a real risk
- adoption friction is likely if the process feels too long or too formal
- limited stakeholder bandwidth makes it harder to move quickly from concept to pilot

One issue that became clearer than I expected is that the initiative has to be positioned very carefully. If it is presented as a large system or another organizational requirement, it will likely feel burdensome. If it is presented as a lightweight first stop that reduces repeated questions, it becomes easier to justify and more appealing to adopt.

Another issue is that, under time pressure, it is tempting to keep revising the concept instead of moving forward with one workable version. Because of that, I have decided that the most useful next step is to stay with the current onboarding and project-clarity direction and treat this mid-semester report as part of locking in that focus.
